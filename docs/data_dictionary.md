# Data Dictionary

This document defines the source fields, database tables, calculated metrics, and final dashboard output used in the Fantasy Football Draft Analyzer project.

The goal of the data dictionary is to make the analytics workflow easier to understand, audit, and extend.

## Source Files

### `data/sample_players.csv`

This file contains the sample player projection pool used by the model.

| Field | Type | Description |
|---|---:|---|
| `player_id` | Text | Unique identifier for each player. Used to join across tables. |
| `player_name` | Text | Player's display name. |
| `position` | Text | Fantasy football position, such as QB, RB, WR, or TE. |
| `nfl_team` | Text | Player's NFL team abbreviation. |
| `projected_points` | Numeric | Sample projected fantasy points for the season. |
| `replacement_points` | Numeric | Estimated replacement-level points for the player's position. |

### `data/sample_draft_picks.csv`

This file contains sample drafted players used to test available-player logic.

| Field | Type | Description |
|---|---:|---|
| `round_number` | Integer | Draft round number. |
| `pick_number` | Integer | Overall or round-level pick number used in the sample draft data. |
| `team_number` | Integer | Fantasy team number that made the pick. |
| `player_id` | Text | Drafted player's ID. Joins to `dim_player.player_id`. |

## DuckDB Tables

### `dim_player`

Dimension table containing player attributes.

| Field | Type | Description |
|---|---:|---|
| `player_id` | Text | Primary key for each player. |
| `player_name` | Text | Player's display name. |
| `position` | Text | Fantasy football position. |
| `nfl_team` | Text | NFL team abbreviation. |

### `fact_player_projection`

Fact table containing player projections and value calculations.

| Field | Type | Description |
|---|---:|---|
| `projection_id` | Integer | Primary key for the projection row. |
| `player_id` | Text | Foreign key to `dim_player`. |
| `season` | Integer | Fantasy season for the projection. Currently set to 2026 in the sample data pipeline. |
| `projected_points` | Numeric | Projected fantasy points for the player. |
| `replacement_points` | Numeric | Replacement-level points for the player's position. |
| `vorp` | Numeric | Value over replacement player. Calculated as `projected_points - replacement_points`. |

### `fact_draft_pick`

Fact table containing players already selected in the draft.

| Field | Type | Description |
|---|---:|---|
| `draft_pick_id` | Integer | Primary key for each draft pick row. |
| `round_number` | Integer | Draft round number. |
| `pick_number` | Integer | Pick number. |
| `team_number` | Integer | Fantasy team number that selected the player. |
| `player_id` | Text | Drafted player's ID. Foreign key to `dim_player`. |

## Final Recommendation Output

### `outputs/draft_recommendations.csv`

This is the dashboard-ready output generated by `src/run_pipeline.py`. It is created from the SQL recommendation view.

| Field | Type | Description |
|---|---:|---|
| `player_name` | Text | Available player's name. |
| `position` | Text | Player's fantasy position. |
| `vorp` | Numeric | Value over replacement player. |
| `scarcity_adjustment` | Numeric | Bonus added when fewer players remain at a position. |
| `roster_count` | Integer | Number of players the current fantasy team already has at that position. |
| `roster_need_adjustment` | Numeric | Bonus added based on current roster need. |
| `recommendation_score` | Numeric | Final score used to rank available players. |
| `recommendation_reason` | Text | Plain-English explanation for why the player is recommended. |

## Metric Definitions

### Value Over Replacement Player, or VORP

VORP estimates how many more fantasy points a player is projected to provide compared with a replacement-level player at the same position.

```text
VORP = projected_points - replacement_points
```

Example:

```text
Projected Points = 215
Replacement Points = 105
VORP = 110
```

A higher VORP means the player is projected to provide more incremental value.

### Available Player Count

Available player count measures how many undrafted players remain at each position.

A player is considered available when their `player_id` does not appear in `fact_draft_pick`.

This is calculated with the SQL pattern:

```sql
LEFT JOIN fact_draft_pick fdp USING (player_id)
WHERE fdp.player_id IS NULL
```

### Scarcity Adjustment

The scarcity adjustment gives a small bonus to positions with fewer available players remaining.

```text
scarcity_adjustment = 10.0 / available_player_count
```

Example:

```text
Available TEs = 9
scarcity_adjustment = 10.0 / 9 = 1.11
```

A lower available player count creates a higher scarcity adjustment.

### Roster Count

Roster count measures how many players the current fantasy team already has at each position.

The current sample model uses `team_number = 1` as the current team.

### Roster Need Adjustment

The roster need adjustment adds a bonus when the current team has not filled a position yet.

```text
0 players at position = +8
1 player at position  = +4
2+ players at position = +0
```

This simple rule helps the model recommend players that fit the current roster, rather than ranking only by raw value.

### Recommendation Score

The recommendation score combines the main decision factors into a single ranking metric.

```text
recommendation_score = VORP + scarcity_adjustment + roster_need_adjustment
```

Players are sorted by recommendation score in descending order.

### Recommendation Reason

Recommendation reason translates the numeric score into a human-readable explanation.

Current reason categories include:

| Reason | Meaning |
|---|---|
| `High VORP + scarce position + roster need` | Strong projected value, limited remaining supply, and position fits the current roster. |
| `High VORP + roster need` | Strong projected value and fills a roster need. |
| `Scarce position` | Position has limited supply remaining, even if projected value is lower. |
| `Depth option` | Player remains available but is not strongly boosted by scarcity or roster need. |

## Current Assumptions

The current project version uses simplified assumptions to keep the model explainable:

* The sample league is a 12-team half-PPR fantasy football league.
* The current team is represented by `team_number = 1`.
* Only QB, RB, WR, and TE are included in the current player pool.
* Player projections are sample values used for portfolio demonstration.
* Replacement points are manually assigned by position in the sample dataset.
* Injuries, bye weeks, strength of schedule, and live draft data are not included yet.

## Data Quality Checks to Consider

Future versions can add checks such as:

* No duplicate `player_id` values in `dim_player`
* No drafted player missing from `dim_player`
* No negative projected points
* No missing positions
* No missing recommendation scores
* Draft picks should not duplicate the same `player_id`
* Every row in the dashboard output should represent an available player only

## Future Data Improvements

Future versions may replace or enhance the sample data with:

* Historical NFL player data
* Fantasy projections from public or licensed sources
* Sleeper API draft and roster data
* Yahoo Fantasy API data
* Bye week data
* Injury indicators
* Average draft position, or ADP
* Historical backtesting results
